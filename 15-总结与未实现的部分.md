## 15.1 全景设计取舍表

每一行都是 mini-sglang 在做的"权衡"，**取舍方向是哪个、代价是什么**。

| # | 决策 | 取舍方向 | 主要代价 | 对应章节 |
|---|------|---------|---------|---------|
| 1 | 多进程而非多线程 | 隔离性（asyncio + GPU 不互相绑定 GIL） | 必须有消息总线（ZMQ）+ 启动开销 | 1 |
| 2 | ZMQ 控制平面 + NCCL 数据平面 | 分工清晰、通信开销低 | 两套通信代码 | 1, 11 |
| 3 | rank 0 单点接 ZMQ 再 PUB 给其余 rank | tokenizer 不需要懂 TP；保证消息顺序一致 | rank 0 多干一份 IO；广播延迟 | 1, 6 |
| 4 | Tokenize / Detokenize 默认共用一个进程（`--num-tokenizer 0`） | 节省一个 tokenizer 文件加载（几百 MB） | 高并发时单进程 CPU 满负 | 1 |
| 5 | KV cache 池存 token-level slot id 在 page_table | `page_table[input_mapping]` 一步拿 slot；attention 接口干净 | page_table 占 page_size 倍内存（int32，开销小） | 4 |
| 6 | RadixCache 默认而非 NaiveCache | 跨请求前缀复用，性能巨大提升 | 实现复杂；split / lock / evict 三个不变量必须维护正确 | 5 |
| 7 | dummy_req + `+1` 多分一行/一页 | CUDA Graph batch padding 干净 | 极小内存代价；写代码时要分清 reqs vs padded_reqs | 4, 8 |
| 8 | Prefill 永远优先 decode | 控制 TTFT 抖动 | Decode ITL 抖动可能上升；并发多时 TTFT 仍可能变长 | 6 |
| 9 | Chunked Prefill 默认 max=8192 | 控制单次 prefill 时长 → decode 不被卡太久 | 长 prompt 总耗时略增（多次 launch） | 6, 10 |
| 10 | Overlap Scheduling 默认开 | 小模型/中等并发吞吐提升 30%+ | 代码复杂；finished_reqs 防御不可去 | 7 |
| 11 | 模型层用全局 `Context._batch` 通信 | 模型层签名干净（`forward(self)`） | 不可重入；测试要小心 | 3, 8 |
| 12 | meta tensor + 流式 weight load | CPU/GPU 内存峰值低 | safetensors 读盘 IO 不能并行 | 8, 13 |
| 13 | pynccl 替代 torch.distributed.nccl | 不破坏 CUDA Graph | 多一份 C++/CUDA 代码维护 | 11 |
| 14 | 自定义 CUDA kernel（store/index/radix）+ tvm-ffi JIT | 性能 + 编译/部署灵活 | 首次启动慢（JIT），需 nvcc + sgl_kernel | 14 |
| 15 | Attention 后端注册表 + HybridBackend | 可插拔 fa/fi/trtllm，prefill/decode 独立选 | 每加新后端要写 5 个方法 | 9 |
| 16 | CUDA Graph 只对 decode 用，桶化 `[1,2,4,8,...]` | 大幅省 launch overhead | 占 1-2 GB 显存；prefill 不能用 | 10 |
| 17 | MoE 用 fused triton kernel + token 排序 | 单 kernel 算所有 expert | block padding 浪费；不支持 EP | 12 |

```

                            │ HTTP / SSE

┌──────────────────────────────────────────────────────┐
│  API Server (asyncio, FastAPI, uvicorn)              │
│   - FrontendManager: ack_map, event_map              │

│   - StreamingResponse 包成 OpenAI chunks             │

              │ ZMQ ipc 4 (TokenizeMsg)
              ▼
┌──────────────────────────────────────────────────────┐
│  Tokenizer / Detokenizer (1 ~ N+1 进程)              │
│   TokenizeManager: text → input_ids                  │
│   DetokenizeManager: token id → 增量字符串            │
└────────────┬───────────────────────────────────────────┘
              │ ZMQ ipc 0 (UserMsg)
              ▼
┌──────────────────────────────────────────────────────────────────┐
│  Scheduler rank 0 ─ ZMQ PUB ipc 2 ─→ rank 1..N-1                 │
│   ├ TableManager: page_table 行号池 + token_pool                 │
│   ├ CacheManager: free_slots + RadixPrefixCache                  │
│   ├ DecodeManager: running_reqs + inflight_tokens                │
│   ├ PrefillManager: pending_list + PrefillAdder                  │
│   ├ run_forever()                                                │
│   │   ├ overlap_loop():                                          │
│   │   │   recv_msg → schedule → forward (engine.stream)          │
│   │   │   process_last_data (上一步)                             │
│   │   └ normal_loop(): 串行版本（ENV.DISABLE_OVERLAP=1）         │
│   └ Engine (per process, per GPU)                                │
│       ├ Model (Qwen3 / Llama / Qwen3-MoE / Mistral)              │
│       │   每层: AttentionLayer + MLP/MoEMLP + RMSNormFused        │
│       ├ MHAKVCache: (2, L, P+1, S, kv_h, dim) 大张量              │
│       ├ AttentionBackend: FA / FlashInfer / TRT-LLM / Hybrid     │
│       ├ MoeBackend: FusedMoe (only)                              │

│       └ GraphRunner: CUDA Graph capture/replay 桶化               │
└──────────────────────────────────────────────────────────────────┘
       │
       │ NCCL / pynccl all_reduce / all_gather (在 forward 里)
       │ gloo barrier / broadcast (CPU 同步消息计数)
       ▼
   各 TP rank GPU
```

## 15.3 与 nano-vllm 对比

如果你之前学过 nano-vllm，下面这张表帮你建立"差异感"：

| 维度 | nano-vllm | mini-sglang |
|------|----------|-------------|
| 代码量 | ~1,200 LOC | ~5,000 LOC |
| 部署模型 | 离线 LLM 类 | 离线 LLM + 在线 OpenAI 兼容 server |
| 进程模型 | 单进程 + SharedMemory + Event 做 TP | 多进程 + ZMQ + NCCL |
| Tokenizer | 主进程做 | 单独进程，可多个 |
| Prefix Cache | xxhash 哈希链，块级别共享 | RadixTree 树结构，token 级别（page-aligned） |
| Attention 后端 | 仅 flash-attn（一种） | fa / fi / trtllm，prefill+decode 独立选 |
| KV cache 索引 | block id（要算 offset） | token-level slot（直接索引） |
| CUDA Graph | 仅 decode，简单的 bucketing | decode + capture buffer + 多 backend 适配 |
| Scheduler | step() 串行 | overlap_loop（双 stream 重叠） |
| Chunked Prefill | 仅第一个 seq 可 chunk | 任意 seq、按 budget 切 |
| MoE | 不支持 | Qwen3-MoE，FusedMoe Triton kernel |
| 自定义 kernel | 不需要 | store/index/radix/pynccl/MoE 五个 |
| 模型支持 | 只 Qwen3 | Llama, Qwen2/3, Qwen3-MoE, Mistral |
| 抽象层 | 几乎没有，所有逻辑直接写 | Manager / Backend / Layer 三层抽象 |

**关键观察**：

1. **nano-vllm 是"教学最小版"**：把 vLLM 的核心思想（PagedAttention + continuous batching + chunked prefill + TP）压缩在最小代码里。读完它 LLM 推理的"基础物理学"就清楚了。
2. **mini-sglang 是"工业小版"**：它假设你要做生产服务（API server + 流式 + abort + 多模型），并且追求性能（overlap、自定义 kernel、多后端）。它的代码不是为了简洁，是为了**正确而高性能**。
3. **学习路径建议**：先 nano-vllm 建立直觉，再 mini-sglang 学工程设计。

## 15.4 与工业 SGLang 对比：缺了什么

mini-sglang ≠ SGLang。SGLang 工业版（[`sgl-project/sglang`](https://github.com/sgl-project/sglang)）有 mini-sglang 没实现的特性：

### 没实现：分布式扩展

- **Pipeline Parallelism (PP)**：把模型按层切到多个 GPU 上。mini-sglang 只 TP，不 PP。
- **Expert Parallel (EP)**：每个 GPU 持有部分 expert。mini-sglang 是 EP=1（每个 rank 全部 expert，TP 切 intermediate）。
- **DP (Data Parallel) for routing**：MoE 在 attention 用 TP、在 MoE 用 EP/DP 的混合策略。

### 没实现：高级调度

- **优先级调度**：高优请求优先 schedule。mini-sglang 是 FCFS。
- **Cooperative scheduling**：多个 server 之间均衡负载。
- **Speculative Decoding**：用 draft model 一次出多个 candidate token，主模型一次验证 N 个——大幅提升单请求吞吐。
- **Constrained Decoding / JSON Mode**：受语法约束的生成。

### 没实现：缓存层级

- **HiCache**（GPU + CPU + Disk 多级 KV cache）：[`base.py:65`](../../python/minisgl/kvcache/base.py) 的 `MatchResult` 有 `# TODO: support HiCache` 注释，给未来留口子。
- **Cross-request KV transfer**：当前 prefix cache 只在同一 GPU 上共享；多机时通过 RDMA 转移 KV。

### 没实现：量化

- **AWQ / GPTQ / FP8 / INT4**：mini-sglang 只支持 fp16/bf16。
- **KV cache quantization**：8-bit / 4-bit KV。

### 没实现：模型多样性

- **MLA (Multi-head Latent Attention)**：DeepSeek 的注意力变种，KV 大幅缩小。`MHAKVCache` 注释里 `# TODO: support other variants (e.g. MLA)`。
- **MoE-Mistral / Mixtral**：只 Qwen3-MoE。
- **多模态视觉模型**：vision_tower 被忽略（只用 LLM 部分）。

### 没实现：可靠性

- **Health check / metrics**：Prometheus 等。
- **优雅升级 / 滚动重启**。
- **日志结构化 / 追踪**：当前是简单的 logger。

### 没实现：性能 micro-optimization

- **ROCm 支持**：仅 NVIDIA。
- **SwiGLU 用更高效 kernel**（mini-sglang 用 sgl_kernel 的 silu_and_mul）。
- **CUTLASS 算子整合**。

> 这不是说 mini-sglang 差——它的目标是**用 ~5000 行讲清楚 SGLang 的核心设计**。少了的这些都是工程化扩展，不是核心思想。

## 15.5 mini-sglang 比 nano-vllm 多出的 "工程心智"

读完 mini-sglang 你应该建立这些 nano-vllm 没教你的"工程心智"：

### 1. 分布式系统的拆边界

把 server / tokenizer / scheduler / engine 各自独立成进程，用消息总线连接——这种拆法在所有大规模推理服务里都有。学完这套就能读 vLLM、TGI、TensorRT-LLM 的源码。

### 2. CPU 与 GPU 的协同

- 哪些数据必须在 CPU（input_ids 累积、metadata 构造）。
- 哪些必须在 GPU（KV cache、token_pool、page_table）。
- 哪些是 CPU 写 → 异步 H2D 给 GPU（`pin_memory + non_blocking=True`）。
- 哪些是 GPU 写 → 异步 D2H 给 CPU（next_tokens_cpu）。
- 哪些是同 stream 串行、哪些是双 stream 并行（overlap）。

### 3. CUDA Graph 的边界

- 什么 op 能 capture（计算 op、collective op）。
- 什么不能（分配、同步、host 端代码）。
- 怎么用 capture buffer + replay 时拷数据避开限制。

### 4. NCCL 的选型

- 默认 torch.distributed.nccl 不友好（隐式同步），自己写 pynccl。
- gloo 用于 CPU 同步，NCCL/pynccl 用于 GPU collective。
- 通信子要 bootstrap（rank 0 创建 nccl_id 广播给其余 rank）。

### 5. 状态机与不变量

`cached_len ≤ device_len ≤ max_device_len`、`extend_len = device_len - cached_len`、`PrefillAdder.lock 之后再次检查 available_size`、`finished_reqs 防双 free`——这些都是必须维护的不变量。系统能跑得正确就是靠它们；改代码时不能破坏。

## 15.6 你能怎么用这套代码

### 1. 学习路径

按本文档顺序读完，每章末检查清单都做一遍——面试 LLM 推理引擎的题（vLLM 怎么实现、prefix cache 怎么做、CUDA Graph 在哪些场景下有问题）应该都能答。

### 2. 改造项目

mini-sglang 的代码量足够小，可以做一些有趣的改造：

- **加 speculative decoding**：在 Engine 的 forward 里加 draft model 的 forward + 验证逻辑。
- **加 PP**：在 Engine 之间加 pipeline 通信。需要把 model.forward 拆成 stages。
- **加 KV offload**：当 free_slots 不够时把最老的 page 换到 CPU 内存。需要扩展 RadixCache 的 evict + 增加 H2D/D2H 路径。
- **加新模型**：3 步——`models/foo.py` 实现、`register.py` 注册、`weight.py` 处理特殊 weight 名。
- **改 schedule policy**：`scheduler.py:_schedule_next_batch` 那个 TODO，换成 decode-first 或 hybrid。

### 3. 读 SGLang / vLLM 源码

掌握 mini-sglang 后，读工业版的速度会快很多——所有核心抽象（Scheduler / KV cache / Backend / Hybrid）都是同一套模型。差别在工程化扩展（量化、HiCache、PP/EP、speculative）和 micro-opt。

## 15.7 总结的总结

mini-sglang 同时是这三样东西：

1. **一个高质量推理引擎**：~5000 行，能跑 OpenAI 兼容 server，性能在中小模型上接近 SGLang。
2. **一份 SGLang 的"骨架文档"**：把工业 SGLang 的设计骨架抽出来，去掉所有"工程化扩展"。
3. **一份学习材料**：你只要把这 15 章的检查清单做完，对 LLM 推理的理解就远超大多数从业者。

**它不教你的**：
- 怎么把模型从 60% 利用率优化到 90%（要看具体 kernel + profile + 改 launch config）。
- 怎么做大规模分布式部署（K8s、autoscaling、负载均衡）。
- 算法本身（量化、长上下文优化、distillation）。

**它教你的**：
- 推理引擎的 5 大矛盾和它们的解法。
- 多进程异步系统的边界划分和通信。
- CPU 调度 + GPU 计算的并行设计。
- 工程不变量的维护（finished_reqs / cache_req / split_at 等）。

读到这里你应该能回答最初的问题：**"为什么我们需要一个推理引擎、它在做什么"**——它在调和"批处理 vs 异质性"、"显存 vs KV"、"调度决策"、"多组件解耦"、"CPU/GPU 协同"这五对矛盾，用 paged attention、prefix cache、phase separation、process topology、overlap scheduling 这些组合招数。

剩下的全是工程细节。

## 15.8 终极检查清单
